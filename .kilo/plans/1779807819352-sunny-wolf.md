# Plan: Подписка на задачи + Email-уведомления о дедлайнах

## 1. Текущая архитектура (ключевые находки)

### Уже существует:
- **Таблица `listener_task`** — M:M, но использует `listener_id INT` (user.id) и `task_id INT` (task.task_id), а не UUID — неудобно для текущего UUID-based service layer
- **`mail_sender`** — SMTP, только восстановление пароля. Имеет Redis
- **Board сервис** — НЕТ исходящих gRPC-соединений (только сервер), есть Postgres + Redis + broker Redis
- **gRPC** — только facade → microservices. Microservices НЕ вызывают друг друга
- **Redis Streams** (`pkg/pubsub/redis`) — используется для real-time WebSocket (board → facade → ws), не для inter-service

### Ограничения:
- `listener_task` использует `listener_id INT`, а вся бизнес-логика работает с `user_link UUID`
- User DB отдельная — board не может сделать SQL JOIN для получения email
- Board сервис не имеет gRPC-клиентов (нужно добавлять)

## 2. Где расположить Worker?

Рассмотрены 4 варианта:

### Вариант A: Worker в board сервисе ✅ (рекомендуется)
- **За:** есть прямой доступ к картам, дедлайнам, подписчикам, Redis
- **За:** один сервис владеет доменной логикой (задачи + дедлайны)
- **Против:** нужно добавить gRPC-клиент к mail_sender
- **Итог:** оптимальный баланс — Board владеет данными, ему и уведомлять

### Вариант B: Scheduler в mail_sender
- **Против:** mail_sender не имеет доступа к Postgres board'а. Пришлось бы дублировать подключение — нарушение изоляции
- **Итог:** ❌ не рекомендуется

### Вариант C: Новый микросервис notification-scheduler
- **Против:** новая деплой-единица, configs, secrets, Helm template, CI/CD — overkill для периодического worker'а
- **Итог:** ❌ избыточно

### Вариант D: Scheduler в facade
- **Против:** фасад не имеет доступа к БД (он только HTTP → gRPC прокси)
- **Итог:** ❌ не имеет смысла

**Решение: Вариант A — Worker внутри board сервиса**

## 3. Как Worker будет работать?

### 3.1 Жизненный цикл

```
board service startup
├── gRPC server (существующий)
├── gRPC client к mail_sender (НОВЫЙ)
└── Scheduler goroutine (НОВЫЙ)
    └── ticker: раз в 1 час (конфигурируемо)
        └── processDeadlines()
            ├── 1. Определить временные окна
            │    └── [now + 7d - 6h; now + 7d + 6h]
            │    └── [now + 3d - 6h; now + 3d + 6h]
            │    └── [now + 1d - 6h; now + 1d + 6h]
            ├── 2. SQL: SELECT задачи в этих окнах
            ├── 3. Для каждой задачи:
            │    ├── Получить подписчиков (user_link) из task_subscriber
            │    ├── Для каждого подписчика:
            │    │    ├── Проверить Redis: "notif:remind:{task}:{user}:{days}" существует?
            │    │    ├── Если нет:
            │    │    │    ├── Получить email из Redis: "user_email:{user_link}"
            │    │    │    ├── Если email найден:
            │    │    │    │    └── gRPC: mail_sender.SendDeadlineReminder(...)
            │    │    │    └── Установить Redis-флаг с TTL = (deadline - now) + 24h
            │    │    └── Если да: пропустить
            └── 4. Логирование результатов
```

### 3.2 Почему polling, а не event-driven?

Задачи с дедлайнами хранятся в Postgres. Варианты отслеживания изменений:
- **CDC (Debezium, pg_logical)** — переусложнено
- **Триггеры при изменении dealline** — можно, но deadline меняется редко, а поддержка триггеров добавляет耦合
- **Polling раз в час** — достаточно для напоминаний за 7/3/1 день. Окно ±6 часов допускает неточность

Polling — pragmatic choice. Добавление event-driven подхода (публикация события при установке/изменении дедлайна) — **опциональное улучшение** на будущее.

### 3.3 Дедупликация

Ключ в Redis: `notif:remind:{task_link}:{user_link}:{days_remaining}`
- Устанавливается перед вызовом gRPC
- TTL = оставшееся время до дедлайна + 24 часа запаса
- Если ключ существует — напоминание уже отправлено
- При изменении дедлайна старые ключи протухнут сами (TTL считается от старого дедлайна)

### 3.4 Получение email пользователя

**Проблема:** User DB отдельный сервис, board не может сделать JOIN.

**Решение — кэширование email при подписке через facade:**

```
HTTP: POST /api/v1/tasks/{task_link}/subscribe
  facede (знает user_link, user_email из сессии)
    └── gRPC: board.SubscribeToTask(user_link, task_link, user_email, user_name)
          ├── INSERT INTO task_subscriber (task_link, user_link) — новая таблица с UUID
          └── Redis SET "user_email:{user_link}" = email, TTL = 30d
          └── Redis SET "user_name:{user_link}" = name, TTL = 30d
```

Facade передает email. Board кэширует в Redis. Scheduler читает из кэша.

**Если email нет в кэше (не подписывался):** пропускаем напоминание. Это ок — дедлайн-напоминания только для подписчиков, а при подписке email гарантированно сохранится.

### 3.5 Новая таблица `task_subscriber`

Вместо использования `listener_task` (с INT FK) — новая таблица:

```sql
CREATE TABLE task_subscriber (
    task_link UUID NOT NULL REFERENCES task(task_link) ON DELETE CASCADE,
    user_link UUID NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now() NOT NULL,
    PRIMARY KEY (task_link, user_link)
);
```

**Почему не `listener_task`:**
- Использует `listener_id INT` (user.id) — а board оперирует UUID
- `task_id INT` — тоже не UUID
- Добавление UUID-колонок в listener_task = миграция с дефолтами
- Новая таблица чище: UUID → UUID, без лишних JOIN-ов

## 4. Коммуникация: board → mail_sender

**Варианты:**

### A: gRPC напрямую (board клиент → mail_sender сервер) ✅
```
board (scheduler) --gRPC--> mail_sender
```
- **За:** просто, синхронно, легко дебажить
- **Против:** board становится gRPC-клиентом (новый паттерн)
- **Требуется:** добавить connection management в board/app

### B: Redis Stream (board публикует → mail_sender подписывается)
```
board (scheduler) --XADD--> Redis Stream --XREAD--> mail_sender
```
- **За:** асинхронно, board не ждет ответа
- **Против:** гарантии доставки сложнее, mail_sender без Postgres не может проверить дубли
- **Требуется:** consumer group, acknowledgment, retry logic

**Решение: Вариант A (gRPC напрямую)**, но с асинхронным паттерном внутри mail_sender (в горутине с ретраями, как уже сделано в `SendRecoveryCode`).

## 5. Почему не RabbitMQ/Kafka?

Обоснование из анализа:
- У вас уже есть gRPC и Redis как инфраструктура
- RabbitMQ/Kafka не нужны потому что:
  - Одно направление данных (board → mail_sender)
  - Нет сложной маршрутизации
  - Нет высокой нагрузки (≤1000 уведомлений/час)
  - Не нужна replay событий (CDC)
  - gRPC + Retry достаточно для надежности
- Kafka была бы полезна при: event sourcing, >10k msg/sec, CQRS, множестве consumer'ов с разной логикой

### 6.3 Board service — Scheduler

#### Новый пакет: `board/internal/notifications/`

```go
// config.go
type Config struct {
    SchedulerInterval time.Duration // по умолчанию 1h
    MailSenderAddr    string
    ReminderWindows   []ReminderWindow
}

type ReminderWindow struct {
    Days      int
    Window    time.Duration // ±6h по умолчанию
}

// reminder_task.go
type TaskDeadlineInfo struct {
    TaskLink    uuid.UUID
    BoardLink   uuid.UUID
    Title       string
    Description string
    Deadline    time.Time
}

// scheduler.go
type Scheduler struct {
    db          cardRepo.DBEngine
    redisClient *redis.Client
    mailClient  pb.MailSenderServiceClient
    cfg         Config
    logger      zerolog.Logger
}

func NewScheduler(db, redis, mailClient, cfg, logger) *Scheduler

func (s *Scheduler) Run(ctx context.Context) {
    ticker := time.NewTicker(s.cfg.SchedulerInterval)
    defer ticker.Stop()
    
    for {
        select {
        case <-ticker.C:
            s.processAllWindows(ctx)
        case <-ctx.Done():
            return
        }
    }
}

func (s *Scheduler) processAllWindows(ctx context.Context) {
    for _, window := range s.cfg.ReminderWindows {
        s.processWindow(ctx, window.Days)
    }
}

func (s *Scheduler) processWindow(ctx context.Context, days int) {
    // 1. Вычислить окно: [now + days*24h - 6h; now + days*24h + 6h]
    // 2. SQL: SELECT task_link, title, description, due_date 
    //    FROM task_actual WHERE due_date BETWEEN $1 AND $2
    //    AND status = false (не выполнена)
    // 3. Для каждой задачи:
    //    a. GetSubscribers(task_link) — user_link[]
    //    b. Для каждого user_link:
    //       - dedupKey = "notif:remind:{task_link}:{user_link}:{days}"
    //       - exists? → пропуск
    //       - email = redis.Get("user_email:{user_link}")
    //       - name = redis.Get("user_name:{user_link}")
    //       - email + name есть? → gRPC mail_sender.SendDeadlineReminder(...)
    //       - redis.Set(dedupKey, "1", TTL)
}
```

## 7. Масштабирование и отказоустойчивость

### При нескольких репликах board service:
- Scheduler запускается в каждой реплике
- Но все пишут в один Redis — дедупликация работает
- Первая реплика, вызвавшая `SET NX`, победит
- Остальные увидят, что ключ уже есть, и пропустят

### При падении board service:
- После перезапуска scheduler продолжит с нового тика
- Redis-флаги с TTL переживают рестарт
- Потерянные тики — не критично (окна ±6 часов)

### При недоступности mail_sender:
- gRPC вернет ошибку, scheduler залогирует
- Напоминание не отправится в этом окне
- Следующий тик через 1ч повторит попытку (пока deadline не прошел или TTL не истек)
